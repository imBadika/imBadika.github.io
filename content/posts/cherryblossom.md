В этот раз мы рассмотрим комнату на THM под названием **CherryBlossom**. Её автор **MuirlandOracle**. Вообще, за его авторством на этой платформе есть несколько комнат. Например, я проходил одну теоретическую на тему эксплуатации уязвимости, связанной с загрузкой файлов в веб-приложениях.

Получилось выудить много полезной информации. Начнем с того, что у нас есть 445 порт, на котором висит SMB. Но, так как я запустил nmap с флагом **-sc**, вдобавок ко всему этому мы видим, что есть возможность анонимного доступа (account_used: guest). То есть, мы можем просмотреть список доступных шар, и даже подключиться к какой-нибудь шаре. Для этого используем smbclient.


Как видим, старый nmap нам не наврал, возможность анонимного подключения к SMB действительно существует. Больше всего нас здесь интересует шара под манящим названием **Anonuymous**. Мы присоединимся к ней и посмотрим, что же она скрывает.

```bash

smb: \> get journal.txt
```

После подключения к шаре Anonymous я нашёл файл с говорящим названием, который сразу же забрал к себе на комп. Если вам интересно содержимое этого файла точно также, как и мне, то спешу вас разочаровать, оно закодировано.

![Содержимое файла](/images/cblossom-journal.webp)

Мы видим здесь кодировку base64. Думаю, для начала стоит попробовать все это привести к "человеческому" виду, прибегнув к использованию утилиты **base64**.

```bash
$ cat journal.txt | base64 -d > result
```

После декодирования содержимого файла к "человеческому" виду прийти не удалось, но самое главное, что содержимое все-таки было декодировано. Учитывая, что автор явно пытается что-то спрятать, я для начала решил проверить, что из себя вообще представляет получившийся в ходе декодирования файл, и получил предсказуемый вывод.

```bash
$ file result         
result: PNG image data, 1280 x 853, 8-bit/color RGB, non-interlaced
```

Получившийся декодированный файл является изображением формата PNG. А это значит, что нам предстоит столкнуться со стеганографией. Кстати, если вам вдруг интересно, то картинка там следующего характера.

![Картинка](/images/cblossom-stego.webp)

Для того, чтобы разгадать секрет картинки, существует много стеганографических тулов. Я решил воспользоваться **stegpy**. И мне даже удалось кое-что добыть.

```bash
$ stegpy result
File _journal.zip succesfully extracted from result
```

К сожалению, это была не последняя искрометная шутка автора комнаты, потому что архив поврежден.

```bash
$ unzip _journal.zip
Archive:  _journal.zip
file #1:  bad zipfile offset (local header sig):  0
```

На самом деле, хорошо, когда в таких случаях до автоматизма доведена привычка использовать **file**, потому что он может сэкономить часы мозгового штурма.

```bash
$ file _journal.zip
_journal.zip: JPEG image data
```

Конечно, фокус с изменением расширения файла не сработал. Перед нами на этот раз нет ни сакуры, ни самурая, но зато в ходе "упорного" мозгового штурма возникла другая идея. Раз уж это на самом деле не картинка, а вылезло это чудо изначально с расширением .zip, может, стоит попробовать поменять магическое число? Помочь с изменением того самого числа может **hexedit**, ну, а помнить его для .zip архива необязательно (но если помните, снимаю шляпу).

![Изменение магического числа](/images/cblossom-hexedit.webp)

Впереди было две новости, по классике, хорошая - это действительно .zip архив, и его теперь можно открыть, плохая - нужен так называемый passphrase.

```bash
$ unzip _journal.zip
Archive:  _journal.zip
[_journal.zip] Journal.ctz password: 
   skipping: Journal.ctz             incorrect password
```

В этом деле я всегда обращаюсь к своему старому товарищу. В большинстве случаев для него это дело нескольких минут. Кто не понял, обратимся к **john**, а ещё точнее, к его дополнению **zip2john**.

```bash
$ zip2john _journal.zip > zip_hash.txt
ver 2.0 efh 5455 efh 7875 _journal.zip/Journal.ctz PKZIP Encr: TS_chk, cmplen=70461, decmplen=70434, crc=0B987D84 ts=0035 cs=0035 type=8
```

Получившийся хэш кормим самому джону и через пару мгновений наблюдаем результат.

```bash
$ john --wordlist=rockyou.txt zip_hash.txt
Using default input encoding: UTF-8
Loaded 1 password hash (PKZIP [32/64])
Will run 4 OpenMP threads
Press 'q' or Ctrl-C to abort, almost any other key for status
[REDACTED]        (_journal.zip/Journal.ctz)     
1g 0:00:00:00 DONE (2026-07-01 06:39) 50.00g/s 409600p/s 409600c/s 409600C/s 123456..whitetiger
Use the "--show" option to display all of the cracked passwords reliably
Session completed.
```

Как оно неудивительно, в этом архиве оказался другой архив.

```bash
$ unzip _journal.zip
Archive:  _journal.zip
[_journal.zip] Journal.ctz password: 
  inflating: Journal.ctz 
```

Кстати, он тоже запаролен. В поисках информации я подался в ближайший браузер и обнаружил, что добытый мною архив является производной приложения Cherrytree. Как говорится, "у страха глаза велики", потому как передо мной на самом деле оказался 7zip архив, это я понял после изучения информации и совета вновь заюзать file (помним про доведение до автоматизма? :>).

```bash
$ file Journal.ctz 
Journal.ctz: 7-zip archive data, version 0.4
```

Но! В этот раз поиграться с zip2john не выйдет. Но, как я узнал, у джона есть ещё один тул, как раз-таки направленный на 7z архивы. Вы даже не представляете как он называется! Не волнуйтесь, сообщаю: **7z2john**. Синтаксис столь же прост, как и название чудо тула.

```bash
$ 7z2john Journal.ctz > 7z_hash.txt
ATTENTION: the hashes might contain sensitive encrypted data. Be careful when sharing or posting these hashes
```

Как видите, программа крайне настоятельно просит меня не раскрывать хэш. Поэтому последую её совету и ни в коем случае не буду этого делать (самые любознательные могут получить его в ходе прохождения этой комнаты на THM). Я же следую далее и вновь обращаюсь к своему товарищу (абсолютно безвозмездно) с просьбой разгадать passphrase. На этот раз времени ушло чуть больше.

```bash
$ john --wordlist=rockyou.txt 7z_hash.txt
```

Поспешим посмотреть содержимое. Открыть данный архив можно с помощью того самого приложеня Cherrytree, я вижу его, а тем-более пользуюсь им, впервые.

![Cherrytree](/images/cblossom-cherrytree.webp)

Внутри было что-то вроде дневника: записки сумасшедшего. Вкратце: чувак съехал с катушек и его терапевт попросила его вести дневник, ведь это может ему помочь совладать с мыслями. Но мы опустим всю трагику этой истории и сосредоточимся на добыче. Первое: автор дневника пишет о том, что ему удалось собрать список паролей, которые никто и никогда не использовал до этого, и их невозможно будет взломать. Естественно, по доброте душевной он оставил список паролей в своём дневнике. Кроме того, в одной из записей он упомянул, что его подруга, **lily**, выбрала себе пароль. Имя у нас есть и список тоже есть. А ещё у нас есть ssh на 22 порту, вот вам и вектор атаки. Кстати, вторая новость: в этом дневнике был флаг.

![FLAG](/images/cblossom-jflag.webp)

Так как у нас есть юзер, есть список, воспользуемся **hydra** для того, чтобы надыбать себе доступ к системе через ssh.

```bash
$ hydra -l lily -P cherry-blossom.list ssh://10.113.182.130

Hydra (https://github.com/vanhauser-thc/thc-hydra) starting at 2026-07-01 07:14:58
[22][ssh] host: 10.113.182.130   login: lily   password: [REDACTED]
1 of 1 target successfully completed, 1 valid password found
```

После томного ожидания получаем пароль, а следовательно, и доступ к машине по ssh.

```bash
$ ssh lily@10.113.182.130


        #####################################
        ##########  Welcome, Lily  ##########
        #####################################

lily@cherryblossom:~$ 
```

Изначально я пытался найти что-то полезное, осуществляя первоначальное перечисление системы: смотрел getcap, искал файлы с установленными SUID, пытался просмотреть sudo -l, но ничего дельного я не нашёл. Поэтому я принялся рыскать по системе, и, на моё, удивление, когда я пошёл по привычному для себя пути в директорию **/var/backups**, я обнаружил там .bak файлы для passwd и shadow. И я бы прошёл мимо, если бы не заметил, что могу читать **shadow.bak**.

```bash
lily@cherryblossom:/var/backups$ ls -la

-r--r--r--  1 root shadow    1481 Feb  9  2020 shadow.bak
```

Ещё изначально, когда я только осматривался в системе, в директории **/home** я заметил существование ещё одного пользователя: **johan**. Да, его хэш есть в том самом **shadow.bak**.

![Хэши](/images/cblossom-johanpass.webp)

На этот раз я решил обратиться к ещё одному своему товарищу **hashcat**. Для него эта задача по плечу. Использовать я решил тот же список, что и для **lily**.

```bash
$ hashcat -a0 -m1800 '[REDACTED_HASH]' cherry-blossom.list
```

После получения пароля я залогинился за **johan** с помощью **su** в сессии за **lily**, затем решил сразу же проверить директорию **/home/johan**, где, как и следовало этого ожидать, нашёл файл **user.txt**, в котором был заветный флаг.

![Флаг юзера](/images/cblossom-userflag.webp)

Конечно, это ещё не все. Не просто так в системе существует **root** юзер, ведь и его флаг нам тоже нужен. Дальше я долго искал хоть какие-то зацепки, куда же двигаться дальше. Прошерудив доступные файлы для чтения для **johan** с помощью **find**, его разрешения и подобные первостепенные вещи, ничего из ряда вон выходящего и не обнаружил. Всё было также чисто, как и у **lily**. В голову даже лезла идея вскрыть хэш в том самом **shadow.bak** для рут юзера, но это занимает неизвестно сколько времени, поэтому от этой идея сразу отступился.

Затем я решил воспользоваться одним тулом, который сканирует систему и пытается найти, для каких экплойтов она уязвима. Если кто не в курсе, тул **Linux Exploit Suggester**. Тул мне показал несколько уязвимостей и даже дал ссылки на PoC для двух из них, именно поэтому на них я и сосредоточился.Я решил использовать PoC для уязвимости CVE-2021-3493. 

![CVE-2021-3493](/images/cblossom-exploit_1.webp)

Кстати, при попытке использовать данный экспойт я столкнулся с небольшой проблемой. Дело в том, что я, как обычно, скомпилировал **.c** файл с помощью **gcc**.

```bash
gcc exploit.c -o file_1
```

Но я не задумывался о том, что атакуемая машина окажется слишком стара по сравнению с моей. Этот файл там просто не хотел запускаться. Поэтому пришлось повторно выполнить компиляцию, но с добавлением флага **-static**. После запуска эксплойта получилась следующая картина.

![Root Flag](/images/cblossom-rootflag.webp)

Как видите, удалось добыть рут флаг.

> **Примечание**
> Как выяснилось позже, когда я ознакомился с райтапом самого автора, его задумка крылась в ином. На самом деле, по его задумке, нужно было воспользоваться другой уязвимостью, которую LSE мне точно также показал (именно между этой уязимостью и CVE-2021-3493 я и выбирал). Я остановился на CVE-2021-3493, потому что после того, как я прочитал информацию об этой уязвимости, и узнал, что её эксплуатация приводит к получению рут прав, я решил дальше не копать, и воспользоваться эксплойтом.
> Автор же подразумеваел в своей комнате эксплуатацию уязвимости  CVE-2019-18634. 

Дело в том, что при использовании sudo нужно привычным образом ввести пароль юзера. Но что непривычно, так это то, что в терминале вместо пустоты в ответ вводимым символам отображаются звездочки.

![PoC](/images/cblossom-poc.webp)

За это отвечает опция pwfeedback. Под вводимый пользователем пароль в оперативной памяти выделяется фиксированная ячейка определенного размера (буфер). То есть, функция должна была строго следить за переполнением этой ячейки. Но нет худа без добра. Умельцы нашли способ сломать данную проверку.

Лишние вводимые данные в поле пароля выходили за границы буфера и начинали затирать соседние участки памяти. И в самом конце всей этой истории находится важная структура: **Return Address**. Это указатель, который по окончании функции ввода пароля диктовал дальнейшие действия. И именно здесь злоумышленники использовали свой код для выполнения.

В целом, комната вышла достаточно интересной. Как я упомянал в начале, я знаком с работами этого автора на TryHackMe. Я надеюсь, что мои комментарии по ходу решения пришлись вам по душе, а самое главное, были предельно понятными. До новых встреч!
